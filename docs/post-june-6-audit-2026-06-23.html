H-1CRITICALSHIPPED 2026-06-23data exposure
apps/api/src/index.ts:950 (GET /products) · apps/api/src/index.ts:5551 (GET /merchants/:id/inventory)

Both endpoints are unauthenticated. Confirmed by reading the route handlers: there is no requireUser/requireRole call. They run via Prisma (service_role) so RLS does not apply. GET /products returns the full Product rows (including internal-only fields source, source_product_id, product_url, extra_data, createdByStoreId) plus the joined images array, paginated, no auth required. GET /merchants/:id/inventory returns StoreProduct rows + full Product include, again no auth, no field whitelist. This is the same CRIT 4 flagged in the merchant-customer display audit on 2026-06-15 — it remains unfixed.

Compounding bug in /merchants/:id/inventory: the filter where: { storeId: id } still uses the dead pre-Phase-2-FINAL StoreProduct.storeId column, and the category query-param uses where.product.category — which doesn't exist (Product has FKs only). The endpoint will return wrong/empty data when those params are used, in addition to leaking what it does return.

Proposed fix:
  1. Add requireRole(req, res, CATALOG_ADMIN_ROLES) to GET /products. Field-whitelist the response (strip source/source_product_id/product_url/extra_data).
  2. Add requireUser (or requireRole) to GET /merchants/:id/inventory + an authorization check (caller is staff of merchant id OR platform admin). Fix the column (branch_id not storeId) and the category filter (FK join, not scalar).
  3. tsc + smoke test in staging; deploy.
Risk if not fixed: anyone with the API URL can scrape your full 140k catalog including supplier IDs and source URLs.

H-3HIGHSHIPPED 2026-06-23order gate — fail-open vs malformed input
apps/api/src/index.ts (POST /order-requests, "Category Status Gate")

The gate builds reqProductIds = requests.flatMap(items.map(it => String(it?.id)).filter(Boolean)) — empty/missing id values are silently filtered out. If the consumer payload omits productId for any item (legacy cart row, malformed retry), that item flows through the gate without being checked. Today the consumer always sends id, so this is latent, not active — but it's a fail-open shape that should be a fail-closed reject.

Proposed fix: if any item lacks id, reject the whole request with 400 INVALID_ORDER_ITEM (don't .filter(Boolean)). Same change in the validateCategoriesEnabled call site.
SHIPPED 2026-06-23. Deployed via the same EB release as H-1. Scope expanded by mandatory line-by-line audit (rule #3): the original H-3 named one fail-open point (missing id); the audit surfaced and closed two more — (2) empty/missing items array on a request flowed through to write an empty order row; (3) item ids that exist on the wire but don't match any DB Product passed silently because Prisma in just returns fewer rows than requested. All three fail-open points closed in one fix: a front-door validator returns 400 INVALID_ORDER_REQUEST, 400 INVALID_ORDER_ITEM, or 400 UNKNOWN_PRODUCT (with offenders list) as appropriate, BEFORE any downstream gate runs. The silent .filter(Boolean) was removed (not commented over), so future readers can't accidentally rely on the bug-masking behavior. Commit a0dd8594.
H-4HIGHRLS — uncategorized products are immune to the toggle
apps/api/scripts/migrate_category_visibility_rls.ts (cat_vis_product / cat_vis_storeproduct)

By design (and reviewed when the spec was approved): a product with vertical_id IS NULL OR category_id IS NULL is treated as "not gated" → always visible to customers regardless of which category an admin toggles off. This was acceptable when the live catalog was 99.85% categorized — but it means the toggle cannot ever hide an uncategorized product, even via a parent category. We discovered the symptom in this session ("Freshly Vadapalli store stayed visible after Fresh Items was disabled"). Today only "Dark green bel" + a tiny tail remain truly uncategorized, so impact is minimal — but if uncategorized SKUs ever creep back in (e.g. a new partner load), category-disable becomes silently leaky for them.

Proposed fix: add a CI/admin alarm (or a daily Sentry breadcrumb) that fails loud if count(*) FROM "Product" WHERE vertical_id IS NULL AND EXISTS (StoreProduct active listings) > 0. Optionally tighten the RLS predicate to deny uncategorized when a "default-deny" admin setting is on. Treat the alarm as a data-hygiene gate before promoting any new bulk loader.
H-5HIGHCODED · OTA HELDcart prune — supabase .in() default limit + fail-open
apps/consumer-app/src/context/CartContext.tsx revalidateCart()

The function calls supabase.from('Product').select('id').in('id', ids) with no .limit(...). supabase-js defaults to 1000 rows; carts will never approach that, so functionally fine — but more importantly, the read is treated as the source of truth: any product whose id is not returned is pruned and a clearAppliedCoupon() fires. Edge case: if supabase quietly returns a partial result (e.g. server clamps the response set, a retry returns a different snapshot), the code happily prunes items the user actually still has. The catch only resets on a thrown error or an error object — not on partial result sets.

Proposed fix: require data.length === ids.length before pruning; if shorter, log a warning + leave the cart alone. This preserves the legitimate "RLS hid 1 of 5" case (data.length < ids.length is exactly what we want to act on) by also checking that the supabase request didn't error and returned its count matches the request, OR — simpler and equally safe — only prune items whose id is in the requested set AND not in the response set, and never auto-clear the coupon unless >=1 cart item actually moves (already the case — keep that contract).
CODED + AUDITED 2026-06-23 (commit 00615139) — OTA HELD per Pranav's call. Two defensive guards added on top of the existing error/throw handlers: (1) bail if the supabase response is not a real array (null/undefined data with no error treated as anomalous); (2) never auto-wipe a non-empty cart — if every item in the cart would be pruned in one go, treat as a transient blip and skip. The "RLS legitimately hid 1 of N" case still works (the intended path). The extreme "every category in your cart got disabled at once" case is now handled by the server-side H-3 Category Status Gate at checkout instead of by silent local pruning — zero false-positive cart wipes is worth the edge-case delay. Will ship in the next consumer OTA push.
H-6MEDIUM/verticals filter — admin signup picker UX
apps/api/src/index.ts GET /verticals + apps/admin-web (admin "Add Merchant" sheet)

The fix made /verticals filter where: { is_active: true }. That endpoint has only one caller in the merchant signup flow (verified). However: if admin-web's "Add Merchant" sheet ever picks a vertical via the same endpoint, an admin would not be able to onboard a merchant in a temporarily-disabled vertical. Verify the admin path uses GET /admin/categories (which returns all, including disabled) and not /verticals.

Proposed fix: grep apps/admin-web/src for any '/verticals' caller; if present, switch it to the admin endpoint or expose a query param ?includeDisabled=true guarded by requireRole.
H-7MEDIUMRLS — service_role caches in tooling
cross-cutting

Any backend service that caches product/vertical/category rows fetched via service_role (e.g. PostgREST tooling, a future ISR/edge cache, analytics ETL) will not respect the RESTRICTIVE RLS. Today the consumer doesn't use such a path — but the design implicitly assumes every customer-facing read goes through the supabase client as anon/authenticated. Future tooling could introduce a leak.

Proposed fix: add a PR-checklist line item: "Any new endpoint that returns Product/Vertical/Tier2Category/StoreProduct to a customer surface MUST go through supabase (anon/authenticated) OR explicitly re-implement the visibility predicate. Document in CONTRIBUTING.md alongside the GRANT rule."

H-9LOWtooling — pooler transaction quirk
various scripts/recat_*.ts encountered P2028

Prisma's interactive transactions over the Supabase pooler frequently throw P2028 ("Transaction not found") when a transaction runs across many sequential queries. Resolved per-script (resolve IDs upfront, sequential single-row updates) but it'll keep biting future migration/data scripts.

Proposed fix: add an internal helper safeBatchUpdate(prisma, model, updates) that resolves ids upfront then executes one $executeRawUnsafe per batch outside any interactive tx. Document the gotcha in ERRORS.md.
