The root cause — one bug, many symptoms

Most of the serious defects trace to one systemic confusion: the API code treats category and vertical as plain text columns on the Product table — but those columns don't exist. The Product table only has category_id / vertical_id (UUID foreign keys). So every code path that reads or writes the scalar names silently fails:

  • the importer writes a phantom category → Prisma rejects every row → 0 inserted, HTTP 200;
  • the catalog grid's Category & Vertical columns are always blank (the API never maps the FK to a name) — this is exactly what you saw in your screenshot;
  • editing a product's category/vertical from the grid fails 100% of the time;
  • bulk-update 500s, the filter falls back to fake data, and the consumer storefront query 500s.

And the whole class of failures is masked by a MOCK_PRODUCTS fallback that serves 3 fabricated products on any DB error — in production, at HTTP 200 — so bugs look like "real but wrong" data instead of errors.

The 3 critical defects

CRITICAL1. The importer inserts 0 rows but reports success

The bulk-import path writes a non-existent Product.category field (index.ts:2379); Prisma throws on every row, the error is swallowed, and it returns HTTP 200. A 141k load would silently land nothing.

Fix: remove the phantom category key; resolve to category_id/vertical_id; add a 1-row smoke test that asserts a row actually exists.

CRITICAL2. The ₹>1000 ÷100 heuristic corrupts prices

Any MRP over ₹1000 is divided by 100 (index.ts:1466) — ₹1,899 stored as ₹18.99. With the Blinkit data 24% of products are >₹1000, and the wrong price flows all the way to customer orders.

Fix: drop the division for the rupee dataset; reject non-numeric/≤0; audit existing rows.

CRITICAL3. Merchant listings point at the wrong key — 26 of 44 orphaned

StoreProduct.storeId is being set to a branch id, and the FK isn't enforced — so 26 of 44 listings are orphaned and only 2 of 4 stores return any inventory at all. (This is why the catalog looks so sparse.)

Fix: key inventory on branch_id with a real FK; backfill the 26 orphans.

Key high-severity defects

HIGHPhantom category/vertical breaks 6 handlers

Grid edits fail; bulk-update 500s; the catalog filter falls through to mock data; the consumer storefront query 500s — all the same scalar-vs-FK root cause (index.ts:2125, 1057, 2463).

HIGHMOCK_PRODUCTS served in production

On any DB error, GET /products returns 3 fabricated products at HTTP 200 with no environment guard (index.ts:1106) — masking real bugs as silently-wrong data. Fix: return 500; delete the mock fallbacks.

HIGHCustomer-facing (confirms the display audit)

The phased "make it solid" plan

Phase 1 — Unblock the load [M]

Importer inserts; failures become visible.
  • Remove the phantom category key; resolve to ids
  • Delete the MOCK_PRODUCTS fallbacks → return 500 on error
  • Fix the MRP ÷100
  • One-row smoke test (assert a row exists)

Phase 2 — Data integrity [L]

No orphaned/uncategorized; edits + filters work.
  • Fix the storeId/branch_id FK; backfill 26 orphans
  • Replace phantom category/vertical across the 6 handlers (use the FK/relation forms)
  • Approve-flow rejects rows with missing taxonomy
  • Fix the audit-log join

Phase 3 — Customer safety [M]

Correct deletes, veg, images, prices before anyone sees Blinkit.
  • is_deleted:false on consumer queries; soft-delete sets active=false
  • Veg tri-state (unknown = no dot); backfill

Phase 4 — Scale [L]

Browse 141k; resilient bulk.

  • Batched approve INSERT; chunked importer with per-batch transactions + row cap

Phase 5 — Polish [M]
