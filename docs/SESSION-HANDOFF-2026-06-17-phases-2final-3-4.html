3. Phase 2 FINAL — StoreProduct.storeId rework

The problem

The Prisma schema declared a StoreProduct.storeId → Store(id) foreign key that did not exist in the database. storeId was being written with a branch id by the merchant write paths. Result: 26 of 44 listings orphaned; only 2 of 4 stores returned inventory; Freshly silently lost 67% of its catalog (33 real listings, only 11 visible). Zero order_items referenced any orphan.

The decision (Pranav): Option B — eliminate, don't patch

Rather than add the missing FK to a vestigial column, we relocated the canonical Store↔branch link to merchant_branches.store_id (the correct domain location) with a real FK, decoupled all code from StoreProduct.storeId, and set up to drop it. "No leaks, no gaps, no errors."

Brainstorm decisions (all verified against the real 165MB / 141,405-row CSV)

7. Admin catalog taxonomy fix

Founder observation: most products in the admin Global Catalog showed blank Vertical/Category. Root cause (a display bug, NOT missing data — all 139,880 rows have vertical+category+subcategory in the DB): the admin dropdowns were populated from hardcoded, stale arrays (11 verticals / 8 categories) predating the Phase 2 taxonomy (15 verticals / 136 Tier2Categories). A dropdown can't display a value not in its option list → correctly-filed products rendered blank.

Fix (6773db05, admin-web — MasterCatalog.tsx): replaced the hardcoded lists with live data fetched from the Vertical + Tier2Category tables (public-read, same source the customer app uses); relabelled columns to match the customer app + ops language — "Category" (= Vertical, 15) and "Subcategory" (= Tier2Category, 136); Subcategory options cascade by the selected Category. Applied at all 8 sites (catalog table, sync-queue table, edit-product modal, bulk menu, filter). Verified with the real tsc + vite build.

9. Operational know-how + gotchas

  • DB migrations: raw SQL via Prisma $executeRawUnsafe in a tsx script run from apps/api (NOT prisma db execute — it chokes on big inserts). Verify with rolled-back transactions.
  • Run scripts: cd apps/api && npx tsx scripts/<x>.ts. Unit tests: npx tsx --test src/<x>.test.ts (built-in node:test; no jest/vitest).
  • ⚠️ admin-web type-check: npx tsc in admin-web runs a stub package, not the real compiler (silently reports "no errors"). Use node_modules/.bin/tsc --noEmit or npm run build (vite build).
  • API deploy: cd apps/api && npm run build && eb deploy — confirm eb status Ready/Green first; commit src/index.ts (dist is gitignored; .ebignore bundles the working dir). Verify with curl https://api.pickatstore.io/health. Every deploy + every migration needs Pranav's explicit per-action "yes" (the auto-mode classifier enforces this).
  • Bulk DB writes: use createMany (one round-trip/batch), never per-row upsert in a loop over the remote DB (see the perf bug, §6).
  • Loader is resumable: re-run load_blinkit_catalog.ts --skip-existing; idempotent on sourceProductId. Flags: --file --dry-run --limit N --batch-size N --skip-existing.
  • Admin dev server: cd apps/admin-web && VITE_API_PROXY=https://api.pickatstore.io npm run dev → http://localhost:3001.
  • Package managers: api/admin-web/merchant = npm; consumer = yarn.
  • Pre-OTA .env protocol (CRITICAL, in CLAUDE.md): flip EXPO_PUBLIC_API_URL to https://api.pickatstore.io, push with --clear-cache, grep the bundle, then revert. Both consumer + merchant .env sit on the local dev IP for day-to-day work.
