1. Executive summary (plain English)

The coupon engine itself, as deployed on the server, is in genuinely good shape. Six independent reviewers verified the money math, replay protection, ledger and counter bookkeeping, database migrations, and backward compatibility with the old app. Nothing in production is leaking money through the new coupon code today.

However, two new blockers stop a clean sign-off:

  1. The consumer app update about to ship has a dead coupon feature. No add-to-cart path ever records the product ID the coupon system requires, so after the OTA every customer's coupon-apply attempt would fail with "Some items in your cart are out of date." The fix is small and well-bounded (a one-line change at each of ~3 add-to-cart callsites). Zero impact today — phones run the old bundle.
  2. A stale compiled file already in production silently breaks two admin features (admin-invite emails never send; merchant KYC "needs info" returns a 500) — unrelated to coupons, but live right now. The corrected build already sits in the working tree; it needs commit + redeploy.

Everything else found is latent edge cases, admin-dashboard polish, dead code, or items already on the deferred queue. The single biggest standing money risk in the whole system — the server trusting the client's claim that a payment happened — predates the coupon project entirely and is now formally tracked.

Bottom line: Phases 1-5 server-side: solid and live. Consumer-side: one small but critical wire missing before OTA. Two quick fixes and the system is ready for the rollout sequence.

5. What was verified CORRECT (positive assurance)

  • Migrations: all 6 applied cleanly; field-by-field parity between schema and live DB for every coupon column, index, and FK. The one failed migration attempt (order_coupon FK type) was properly rolled back and re-applied.
  • Pricing is fully server-trusted: cart totals recomputed from StoreProduct prices (client prices ignored); percentage-with-cap, flat, both BOGO modes, and the final clamp all independently re-derived and confirmed.
  • Token chain is sound: HMAC-SHA256 with constant-time comparison, expiry + userId binding, exact product-set + quantity-multiset matching — forged, cross-user, stripped, or padded-cart tokens are all closed off.
  • Phase 5 money chain is exact end-to-end: largest-remainder allocation sums precisely; the client consumes the server's signed split verbatim; the ledger accumulates per-slice.
  • Counter/ledger symmetry holds on every traced path: counters increment exactly once per cart and reverse correctly on cancel, including the atomic in-statement partial reversal.
  • Fingerprint dedup: ON CONFLICT target exactly matches the live prod index; per-validation jti makes cross-checkout collisions impossible.
  • Cancel integrity: atomic status compare-and-set makes double-cancel reversal impossible; Razorpay refund fires only after CANCELLED commits.
  • Backward compatibility traced against the ACTUAL deployed consumer bundle: it can hit no coupon-related errors; nothing money-bearing is disabled in the current flag state.
  • Data-layer security verified live: coupons + redemptions unreachable via Supabase REST (RLS + zero grants); three-way RBAC parity (admin-web ↔ API lib ↔ endpoints).
  • Both projects compile clean; prod ledger tables hold 0 rows, so no live data is affected by any finding.
  • The storeProductId round-trip (cart ↔ Supabase) is correctly built on both ends — only the single upstream producer is missing, which is why blocker #1 is a small, bounded fix.

6. The 2 new blockers

CRITICAL · N1 Consumer coupon feature is dead-on-arrival for the pending OTA — no add-to-cart path sets storeProductId
apps/consumer-app/src/screens/StorefrontScreen.tsx L402-413 · FavoritesScreen.tsx L101-111 · hooks/useProducts.ts L72

The Phase 4/5 flow is inert end-to-end: nothing produces CartItem.storeProductId, so the strict guards make every apply attempt fail with "Some items in your cart are out of date." The Supabase round-trip is correct but persists only null. Zero prod impact today (phones run the old bundle) — but the pending OTA would ship a dead feature. This deferred producer work was tracked nowhere — the audit's most important catch.

Fix: pass storeProductId: product.id in StorefrontScreen's handleAdd (id IS StoreProduct.id on the useProducts path) and FavoritesScreen (already keyed by store_product_id); then run the e2e apply test. Now tracked as an explicit OTA gate in forlater.md.

HIGH · N2 Stale committed email.service.js breaks two admin features in prod RIGHT NOW (outside coupon scope)
apps/api/dist/services/email.service.js vs src/services/email.service.ts — deployed dist/index.js calls sendStoreNeedsInfoEmail / sendAdminInviteEmail which the committed service file doesn't export

Live effects today: admin-invite emails silently never send (error swallowed — endpoint reports success), merchant KYC "needs info" 500s after the status update commits, email test-send 500s. Coupon paths unaffected. The corrected rebuilt files already sit as uncommitted working-tree changes.

Fix: rebuild, commit the regenerated dist, redeploy (with deploy confirmation). Longer term: a predeploy guard that fails when tsc output differs from committed dist.
