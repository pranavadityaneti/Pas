Admin Category Enable/Disable — Design Spec

2026-06-19 · branch feat/consumer-global-config-wiring · status: design approved, awaiting spec sign-off

In plain English

Admins get a Categories tab in the dashboard where each category (and its subcategories) has a single on/off toggle. Flipping one off disables it platform-wide: customers stop seeing the category and all its products everywhere (instantly, even on installed apps), merchants can't list new stock in it (and see it as "paused"), and any store left with nothing to sell quietly drops off discovery. Nothing is deleted — turning it back on reverses everything.

Locked decisions

#Decision
D1Scope: toggle both Categories (15 Verticals) and Subcategories (136 Tier2Categories).
D2Single coupled toggle per category/subcategory — disable = platform-wide (customer + merchant together). No separate customer/merchant toggles in v1.
D3Customer side takes effect immediately on live apps via RESTRICTIVE Postgres RLS (no app update).
D4Full hide: the category AND all its products vanish from customer browse, search, and store pages.
D5Admin home: a new "Categories" tab inside the existing Master Catalog module.
D6Merchant coupling: listing API rejects new listings in a disabled category (instant); existing listings parked + "Paused by platform" UX (rides next merchant app update).
D7Stores: no "offline" switch — the existing hide-empty-stores discovery rule is made category-aware, so stores with nothing visible auto-drop and auto-return.

Architecture

1. Data model

Add Vertical.is_active Boolean @default(true) (migration, additive). Tier2Category.active Boolean @default(true) already exists. Index Vertical(is_active).

2. Customer side — RESTRICTIVE RLS (the "instant on live apps" engine)

Existing permissive public-read SELECT policies stay (they're OR'd, can't restrict). Add RESTRICTIVE SELECT policies for anon + authenticated (AND'd onto the permissive ones). service_role (admin API) bypasses RLS entirely → admin still sees/manages everything.

-- Vertical: visible only if active
CREATE POLICY cat_vis_vertical AS RESTRICTIVE FOR SELECT TO anon, authenticated
  USING (is_active = true);

-- Tier2Category: active AND parent vertical active
CREATE POLICY cat_vis_tier2 AS RESTRICTIVE FOR SELECT TO anon, authenticated
  USING (active = true AND EXISTS (SELECT 1 FROM "Vertical" v WHERE v.id = vertical_id AND v.is_active));

-- Product: vertical active AND (no subcategory OR subcategory active)
CREATE POLICY cat_vis_product AS RESTRICTIVE FOR SELECT TO anon, authenticated
  USING (
    (vertical_id IS NULL OR EXISTS (SELECT 1 FROM "Vertical" v WHERE v.id = vertical_id AND v.is_active))
    AND (category_id IS NULL OR EXISTS (SELECT 1 FROM "Tier2Category" t WHERE t.id = category_id AND t.active))
  );

-- StoreProduct: its Product must be visible (airtight regardless of join style)
CREATE POLICY cat_vis_storeproduct AS RESTRICTIVE FOR SELECT TO anon, authenticated
  USING (EXISTS (SELECT 1 FROM "Product" pr
                 WHERE pr.id = "StoreProduct"."productId"
                   AND (pr.vertical_id IS NULL OR EXISTS (SELECT 1 FROM "Vertical" v WHERE v.id = pr.vertical_id AND v.is_active))
                   AND (pr.category_id IS NULL OR EXISTS (SELECT 1 FROM "Tier2Category" t WHERE t.id = pr.category_id AND t.active))));

Verified: customer product reads use Product!inner joins, so these cascade. service_role is unaffected (bypasses RLS), so admin + the merchant API (which reads via service-role per Phase 8/9) keep full access. Apply via $executeRawUnsafe tsx scripts; verify with rolled-back tests (disable → anon hides it + products + listings; re-enable → reappears; service-role unaffected). 3. API — admin (role-gated) + merchant guard

GET   /admin/categories                  → all Verticals + nested Tier2Categories
                                            with active flags + product counts (service-role).
PATCH /admin/categories/vertical/:id     { isActive }   → toggle a Vertical (audit-logged).
PATCH /admin/categories/subcategory/:id  { active }     → toggle a Tier2Category (audit-logged).

Merchant guard (D6): in POST /merchant/store-products/configure, before writing, reject any item whose Product's vertical (or subcategory) is disabled → 403 { error: 'CATEGORY_DISABLED', offenders:[…] } (same shape as the FSSAI/MRP guards). Instant; no merchant app update needed for the block.

4. Discovery — category-aware empty-store hiding (D7)

Update get_nearby_stores (the RPC from Phase 3 Item 4): its EXISTS(active, non-deleted StoreProduct) check additionally requires the product's vertical + subcategory to be active. A store with no visible products auto-drops; multi-category stores stay. Applied via a tsx migration script; verify with a rolled-back disable.

5. Admin UI — "Categories" tab in Master Catalog

New tab listing the 15 Verticals as rows, each with a toggle + a product count, expandable to its Tier2Categories (each a toggle + count). Disable shows a confirm dialog ("This hides the category and all its products from customers and blocks new merchant listings."). Calls the PATCH endpoints; optimistic update + refetch. Reads via the admin API (service-role → sees all incl. disabled).

6. Customer app — cart guard (small)

RLS does the hiding with no app change. One defensive guard: if a cart item's category is disabled mid-session, drop/flag it at cart load + block its checkout (a small consumer change; rides the next OTA — RLS already prevents NEW additions, so this only covers the rare in-flight cart).

Edge cases

CaseBehaviour
Multi-category store, one category disabledOnly that category's products hide; store stays live for the rest.
Single-category store, its category disabledStore has nothing visible → auto-dropped from discovery → auto-returns on re-enable.
Merchant tries to list in a disabled categoryAPI 403 CATEGORY_DISABLED; existing stock parked, not deleted.
Customer has a disabled-category item in cartDropped/flagged at cart load; checkout blocked (consumer guard).
Admin / merchant management of a disabled categoryUnaffected — both read via service-role; admin manages, merchant sees their parked stock.
Disable a subcategory but not its parentOnly that subcategory's products hide; the parent category + other subcategories stay live.

Testing

  • RLS rolled-back tests: disable a vertical → anon SELECT on Vertical/Tier2Category/Product/StoreProduct hides it + its products + listings; service-role still sees all; re-enable → reappears. Same for a single subcategory.
  • API: GET returns all (incl. disabled); PATCH toggles + audit-logs; merchant configure → 403 on a disabled category.
  • Discovery: a single-category store with its category disabled drops from get_nearby_stores; a multi-category store stays.
  • EXPLAIN ANALYZE a representative product read with the restrictive policies (perf at 140k).
  • Admin UI: toggle round-trip + confirm dialog.

Rollback

Drop the 4 RESTRICTIVE policies + revert get_nearby_stores + remove the merchant guard + revert the admin tab. Vertical.is_active column is additive (harmless if left). No data loss at any point.

Out of scope (later)

  • Two-toggle split (customer-hidden vs merchant-allowed) for the "merchants prep inventory before customers see it" case (e.g. pharmacy pre-licence).
  • The "Paused by platform" badge styling in the merchant app is best-effort UX (the *block* is server-side; the badge rides the next merchant build).
  • Category reordering / icon / banner management in the same tab.
