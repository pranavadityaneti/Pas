Merchant Signup + Admin Approval — Audit

2026-06-14. Two-track read-only code audit. (1) Is the merchant signup wizard fully wired, what's pending? (2) Does the admin approval UI surface the new signup fields? Legend: wired/shown partial missing/stubbed · severity HIGH MED LOW.
Headline: The signup wizard works end-to-end for the money + KYC-document path (a merchant can register, upload all docs, pay, submit — all persisted). The admin KYC screen does show the uploaded documents + numbers + bank (a reviewer can actually verify). But the legal layer is hollow: Aadhaar eSign is a 1.5-second fake, and the agreement/consent acceptance is never saved to the database at all — so the admin approves with zero recorded proof the partner agreement was signed. Plus the newer fields (designation, store hours, agreements, extra branches) aren't shown in the review screen.

Part 2 — Admin approval (KYC review)

The real approval screen is KYCQueue.tsx (a read-only drawer MerchantDetailsSheet.tsx also exists). Approve/reject/needs_info → POST /merchants/:id/kyc-decision (OPERATIONS role) → sets kyc_status + emails the merchant. This flow is solid.

What the reviewer CAN see (good)

shown Document images in a zoomable signed-URL viewer: PAN, Aadhaar front/back, GST, MSME, FSSAI certs, store photos. Plus owner/email/phone, city/address, all KYC numbers (PAN/Aadhaar/GST/MSME/FSSAI), turnover, bank beneficiary/account/IFSC. So a reviewer can genuinely verify documents — the core KYC job works.

Recommended priority order

  1. X1 — Agreement/eSign persistence (HIGH, legal). Add a merchant_consents table; have /auth/merchant/draft (or finalize) record accepted docs + versions + timestamp + IP + (real) eSign txn ids; surface a "Legal/Consent" block in KYCQueue. Note: real eSign needs Digio (Task #33) — but persisting the acceptance + doc versions should not wait for Digio.
  2. X2/X3 — Data-integrity fixes (MED). Resolve/show the vertical name (drop the dead category ref); unify the PAN column to one name. Quick, prevents blank/missing fields in review.
  3. Surface the new fields in KYCQueue (MED): designation, branches, bank_accounts, (consent block from #1).
  4. Store hours at signup (MED) — your D1 decision; also fixes the consumer "store always open" issue and the 25 NULL-hours stores.
  5. X4/X5/X6/X7 (MED/LOW): needs_info in the type + a queue view; verify the merchants RPC returns doc columns; persist the reviewer checklist for audit; signed URLs in the details sheet.
  6. Aadhaar eSign / Digio (HIGH for go-live, but vendor-blocked) — Task #33.
