Phase 3 — Returns & Exchanges move to Admin

2026-06-25 · part of "refunds are admin-only; merchant app is display-only" · Pranav decision
In plain English. Today, a merchant approves or rejects a customer's return/exchange request from their app, and approving a return instantly refunds the customer. You've decided that returns should be the admin's job — the admin reviews and decides every return/exchange and issues the refund, and the merchant only sees what happened. This spec is the plan to move that capability from the merchant app to the admin dashboard safely — building the admin side first so no return ever gets stuck with nobody able to act on it.

1) How returns work today

  • Customer submits a return/exchange → an OrderIssue row is created (status='PENDING', type='return'|'exchange', reason, photos[], refundAmountInr, slaDueAt).
  • Merchant decides — merchant app settings/returns.tsx → GET /orders/issues/inbox (their pending list) + PATCH /orders/:id/issue/:issueId ({decision:'APPROVED'|'REJECTED'}). Auth = userCanManageOrderStore. Approve a return → real Razorpay refund + order → RETURN_APPROVED/REFUNDED. Approve exchange → EXCHANGE_APPROVED. Reject → RETURN_REJECTED/EXCHANGE_REJECTED.
  • SLA safety net — processOrderIssueSla cron auto-approves any PENDING issue past slaDueAt (24h) and refunds (tryRazorpayRefund) so returns never hang on merchant inaction.
  • Customer notifications — RETURN_DECISION/EXCHANGE_DECISION fire on decision (incl. refund amount).
  • Admin today: none — admin-web has no returns/issues UI. The admin only has Refunds & Disputes (which issues refunds on cancelled orders).

2) Target model

ActorBeforeAfter
Customersubmits return/exchangeunchanged — submits the same way, still gets decision + refund notifications
Merchantdecides + (approve) refundsdisplay-only sees the request + its status/decision; no approve/reject, no refund
Adminnothingowns it reviews every return/exchange, approves/rejects, and issues the refund

3) The one rule — admin-capability-first

Never strip the merchant's decision power before the admin has it, or pending returns get orphaned (merchant can't act, admin can't yet) and customers wait indefinitely. So: build + deploy + verify the admin returns capability first; only then convert the merchant inbox to display-only (OTA). The order of the phases below enforces this.

4) The build, by surface

API add restrict

  • New: GET /admin/issues (requireAdmin) — list OrderIssue rows across all stores, filterable by status (default PENDING), with order + customer + store + reason + photos + refundAmountInr. Mirrors the merchant inbox shape but admin-scoped + global.
  • New: PATCH /admin/orders/:id/issue/:issueId (requireAdmin) — admin approve/reject. Reuse the exact existing decision logic (the body of the current PATCH /orders/:id/issue/:issueId: status flips, Razorpay refund on return-approve, clawback, RETURN_DECISION/EXCHANGE_DECISION + REFUND_INITIATED notifications) — only the auth changes from userCanManageOrderStore → requireAdmin. Factor the shared logic into one helper called by both, then retire the merchant route (next bullet).
  • Restrict: PATCH /orders/:id/issue/:issueId (merchant) → remove the decision capability. Options: (a) return 403 "returns are handled by the platform"; or (b) delete the route. Recommend 403 first (defensive — stale app builds can't sneak a decision through), delete later.
  • Keep: GET /orders/issues/inbox (merchant) — but it becomes a read-only feed (the merchant app no longer renders action buttons).
  • SLA cron — see Decision D1 (keep as a platform safety-net vs remove).

Admin-web new module

  • Returns & Exchanges queue — a new screen (sibling to Refunds & Disputes; gated by an RBAC permission e.g. returns.resolve). Lists pending issues from GET /admin/issues: order #, customer, store, type, reason, photos, refund amount, age vs SLA.
  • Decision UI — Approve / Reject (with reason) per row → PATCH /admin/orders/:id/issue/:issueId. Approve-return shows the refund amount + confirms it will issue a real Razorpay refund. Resolved rows show the decision + refund id.
  • Surface in Sidebar + SupportHome, behind the permission.

Merchant app display-only (OTA, LAST)

  • settings/returns.tsx — remove handleDecision + the Approve/Reject buttons + the "issue a Razorpay refund" alert. Keep the list as a read-only view: each request shows its status (Pending review / Approved / Rejected) + refund status when present. Add a small note: "Returns are reviewed by the PAS team."
  • No other merchant refund entry points remain (order-reject already stripped in Phase 1).

Notifications — unchanged for the customer

RETURN_DECISION/EXCHANGE_DECISION + REFUND_INITIATED already fire inside the decision logic; reused as-is, now triggered by the admin. The customer experience is identical — they just don't know (or care) that an admin decided instead of the merchant.

7) Related pending (same "merchants don't refund" model)

  • Phase 2 — Booking cancel (bookings.tsx): "This will refund ₹X" on cancel. Needs the same treatment — merchant cancels (no refund) → admin refunds. Smaller; verify the admin queue surfaces cancelled bookings. Queued.
  • Paid-cancel copy: when a paid order is rejected, the ORDER_CANCELLED notification doesn't yet say "a refund will be processed." Small API copy polish so customers aren't anxious between cancel and admin refund.
  • Dead endpoint cleanup: the legacy simulated POST /orders/:id/refund is now unused (Phase 1 removed its only caller) — delete it.

8) Deploy + rollback

  • Vehicles: 3a/3e → EB · 3b → Vercel (admin-web, auto from main) · 3d → EB + merchant OTA.
  • Rollback: 3a/3b are additive (revert commit + redeploy; no data migration). 3d is the only behavior-removal — revert restores the merchant decision route + buttons. The SLA cron remains a backstop throughout, so even mid-migration no return strands.
