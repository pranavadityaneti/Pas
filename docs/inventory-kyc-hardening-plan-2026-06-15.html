Inventory + KYC Hardening — Phase-wise Plan

Pick At Store · 2026-06-15 · read-only audit (8 agents) → 16 phases across 3 tracks · no code written yet

URGENT — independent of the data purchase. Every catalog/product write endpoint (/catalog/sync/approve, /products/bulk-delete, the Apify webhook, all product mutations — index.ts lines ~1500–2420) is completely unauthenticated — verified: zero requireAdmin/requireRole/requireUser calls in that range. Anyone on the internet can inject or bulk-delete products today, and /products/bulk-delete cascades into StoreProduct (merchant inventory). This is Phase 1 and should land regardless of whether you buy the Zepto data.
Verdict. The inventory module is not load-ready for 172k SKUs — and the gap is structural, not cosmetic. Foundations are sound (source_product_id is genuinely unique → idempotent re-runs; the approve path validates taxonomy; null-byte sanitization exists). But the working surface is thin: unauthenticated writes, a price heuristic that corrupts every >₹1000 product, a category mapper that dumps ~25–35 of 40 categories into a non-existent "General" bucket, the one flat-file endpoint dead-on-arrival, no sibling model, and ingestion that loads the whole file into memory + does 172k serial DB calls. The KYC screen works but keeps no decision audit trail, can't render PDF documents, hides several submitted fields, and crams decisions into a 340px column. The plan below sequences the must-fix-before-loading items first.

Must-fix before the 172k load

  1. SECURITYAuthenticate every catalog/product write endpoint (Phase 1) — verified zero auth guards in index.ts 1500–2420.
  2. DATAReplace mapCategory with a data-driven CategoryMapping table (Phase 2) — else most of 172k lands uncategorized + in the wrong vertical ('General' isn't a real category → category_id = NULL).
  3. MONEYDelete the MRP > 1000 ? /100 heuristic for the flat-file path (Phase 5) — the dataset is already in rupees, so every genuine ₹1,200 product silently becomes ₹12.
  4. SCALEBuild a streaming, chunked, resumable bulk-load CLI that bypasses the sync-queue + the synchronous HTTP handler (Phase 5) — the current paths can't finish 172k (in-memory load + 172k serial awaits inside a 60s transaction).
  5. DATALand the additive variantGroupId columns before the load (Phase 3) — else the seller's sibling linkage is permanently lost and can only be reconstructed later from unreliable name heuristics.
  6. PLUMBINGReconcile the StoreProduct migration drift (Phase 4) — schema declares branch_id/is_deleted/new unique key but no migration adds them; the Phase 2/3/7 migrations would apply onto a history that doesn't match prod.
  7. CORRECTNESSDon't stage 172k into the review queue UI — GET /catalog/sync/queue is an unbounded findMany rendered un-virtualized; it freezes the admin tab. The load goes through the CLI, not the queue.

Sibling / variant verdict

Not supported today. There's no parent/variant-group concept anywhere — Product has no variant_group_id; the only "variant" is StoreProduct.variant, a free-text per-listing string. Worse, the merchant catalog-picker's cleanName dedup actively collapses every same-base-name product to one arbitrary pack size, and a modal fabricates fake variants from hardcoded ratios (offering 250ml/500ml/1L for a 5kg atta).

Minimal, non-breaking model: add two nullable columns to Product — variantGroupId (all siblings share one UUID) + variantLabel ("500 ml", "Pack of 6") — plus variant_group_id on SyncQueue. NULL = standalone flat SKU = exactly today's behavior, so nothing breaks. Derive the group UUID deterministically from the seller's parent key (uuidv5) so re-runs are idempotent.

Phasing (important): schema column FIRST (before the load), then ingest-flat-with-linkage, then merchant grouping, then the consumer variant-selector card LAST (fully deferrable — old single-card rendering stays correct while variantGroupId is NULL). You can load flat with zero schema change, but you'd permanently lose the grouping signal unless you capture it during that same load.

KYC UX — current problems & proposed shape

Captured at signup but never shown to the reviewer: business profile (is_veg, cuisines, restaurant type, turnover), bank name, MSME/FSSAI (conditional), the drawn signature image, and any prior decision history. Can't verify documents: the viewer renders everything as an <img>, so PDF certificates show broken and there's no rotate / fit-to-width / side-by-side compare. No accountability: nothing records who approved/rejected a merchant.

Proposed: keep the 3-pane shell but rebuild each pane — a triaged, searchable, oldest-first queue; a content-type-aware document viewer with compare mode; and a widened decision pane grouped into Business Profile / Identity & Compliance / Bank / Legal & Consent, with masked-reveal on Aadhaar+bank, a checklist that names the value being verified ("Name on PAN matches owner: Ravi Kumar"), an Approve confirmation summary, and an audit row on every decision. Served from one endpoint.

Open questions (gate specific phases)

  • Sibling linkage format — does the seller's file have an explicit, stable parent-id column, or is grouping only implied by (brand + base name)? The whole sibling track depends on this. (ask on/after the call)
  • File format — CSV or xlsx? The Phase 5 CLI must stream CSV (xlsx is fully in-memory at 172k).
  • 451 → taxonomy — do we map onto existing categories, or create new Vertical/Tier2Category rows (Paan Corner, Pet Care, Stationery…)? Gates Phase 2 seed quality.
  • Unmapped rows — hold in NEEDS_MAPPING (recommended) or hard-reject from the load?
  • Consumer sibling UX — single card + selector now (Phase 13), or separate cards acceptable for v1?
  • Signature privacy — render the drawn signature inline in the KYC panel, or keep the signed PDF canonical?
  • Job runner — is there a background worker the Phase 5 load can hook, or does it run as a one-off CLI against prod?
