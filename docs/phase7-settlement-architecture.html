1. Executive summary (plain English)

Phase 7 builds the system that answers, every week, for every merchant: "how much do we owe you, and can we prove it?"

The formula: Net payout = what customers paid for the merchant's completed orders + platform-funded coupon discounts we owe back − our commission (per category/tier, per the attached sheet) − clawbacks from earlier weeks (refunds after payout).

Three design principles drive everything below:

  1. Ledger, not live math. When a week closes, we compute once and freeze an immutable settlement record. A merchant's "week of June 1" payout never silently changes in July. Later refunds become explicit deductions in a later week — never a rewrite of history.
  2. Commission rates are data, not code. The sheet itself shows rates vary by category, by order type, and by merchant turnover tier — and that they're still being decided (two tiers are blank). So rates live in an admin-editable table. The two conflicts found in the sheet (§4) don't block the build: founders can settle them and an admin updates a row, no code change.
  3. Calculate now, move money later. Razorpay X rejected our business type, and a payout vendor (Cashfree/Open/Decentro) is not yet selected. Phase 7 computes, records, and displays everything, with a manual "Mark as paid" step. Automated bank transfers become Phase 7b once a vendor is chosen — zero rework needed.

What we need from founders: only §4/§5 — resolve two rate contradictions in the commission sheet, fill in (or default) two blank F&B tiers, and confirm what amount commission is charged on. The build starts now regardless; these arrive as configuration.

3.3 How the engine implements this

  • CommissionRule table (admin-editable): category, order-type qualifier (pickup/dining/any), turnover tier (nullable), rate %, effective-from date. The cycle-close job picks the most specific matching rule per order. Rate changes take effect from the date set — past settled cycles are untouched.
  • Merchant fields (new, admin-set): commission category (derived from their vertical, overridable) and turnover tier (F&B only). Today no turnover data exists in the system, so tier is a manual admin assignment at onboarding/review.
  • Per-order resolution: the order's order_type (pickup vs dine-in — already stored per order) selects pickup/dining rates. This interprets "5% pickup / 7% dining" as per order, not per merchant — see open question FQ-3.

6. Architecture — the settlement ledger

6.1 Why a ledger (and not on-demand math)

If payouts were recomputed live, a merchant's "week of June 1" number could differ in March because an order got refunded later — an audit nightmare and a trust killer. Instead: compute once at cycle close, freeze it, never touch it again. Corrections are new entries, not edits. This is double-entry thinking applied to marketplace payouts, and it matches the no-temporary-patches standard the whole coupon project has been held to.

6.2 Data model (new tables)

SettlementCycle          one row per (merchant, week)
  id, merchant_id, period_start, period_end (IST week)
  status: OPEN → CLOSED → PAID            (forward-only)
  gross_sales              Σ customer-paid totals of eligible orders
  commission_base          Σ per-order commission base (per FQ-4 rule)
  commission_amount        Σ per-order commission (rate resolved per order)
  coupon_reimbursement     Σ PLATFORM-funded coupon slices       (credit)
  coupon_absorbed          Σ MERCHANT-funded slices              (info only)
  clawback_amount          Σ clawback line items applied this cycle
  gst_on_commission, tcs_amount            (reserved, 0 until FQ-6)
  net_payout               gross − commission − clawbacks + reimbursement (− reserved taxes)
  closed_at, paid_at, paid_by, payment_reference (manual for now)

SettlementLine           one row per order per cycle (the receipts)
  cycle_id, order_id, kind: SALE | COUPON_REIMBURSEMENT | CLAWBACK | ADJUSTMENT
  amount, commission_rate_used, commission_rule_id, note

CommissionRule           admin-editable rate matrix (§3.3)
  category, order_type (PICKUP|DINING|ANY), tier (nullable), rate_pct,
  effective_from, provisional (bool — flags §4 defaults)

Merchant additions       commission_category, turnover_tier (F&B),
                         settlement_hold (bool, manual freeze)

6.3 The weekly close (idempotent job + manual trigger)

  1. Select eligible orders for the week: status COMPLETED or RETURN_REJECTED, payment verified against Razorpay (§8), not in the anomaly set, not already settled in a prior cycle (each order settles exactly once — enforced by a unique constraint on SettlementLine.order_id for kind=SALE).
  2. Resolve each order's commission rule (category + order type + tier, most specific wins) and write a SALE line.
  3. Add COUPON_REIMBURSEMENT lines from each order's platform-funded snapshot slice (the per-store slices Phase 5 made exact).
  4. Pull pending CLAWBACK entries (orders refunded after their cycle was paid) into this cycle.
  5. Freeze the cycle: status CLOSED. Anomaly/held orders roll forward to the next week's candidate set automatically — money is delayed, never lost.
  6. Admin reviews in SettlementsManager (currently mock data — gets rewired to this), clicks Mark as paid after making the bank transfer manually, recording a payment reference. Phase 7b replaces this click with a payout-vendor API call.

6.4 What each party sees

  • Admin (SettlementsManager): per-merchant weekly rows — gross, commission, reimbursements, clawbacks, net, status, held-order count; drill-down to the per-order lines; mark-as-paid flow. FinanceHub KPI: total platform coupon spend + total commission earned per week.
  • Merchant (Earnings tab): a settlement history list — "Week of June 1: ₹14,520 · PAID" — with the same line-item breakdown. The current "2% fee simulation" estimate gets replaced by real numbers derived from their actual commission rules.

8. Prerequisite: server-side payment verification

The single biggest standing money risk in the system predates the coupon project: the server currently trusts the app's claim that an order was paid (paid: true + a payment id in the request). Until now that risk was bounded — wrong data, no automated money. Phase 7 is the moment the data starts driving real payouts, so verification graduates from "queued hardening" to a prerequisite:

  • At order creation: verify the payment id against Razorpay's API — captured, amount matches, not previously used. Orders that fail go to the anomaly bucket (created, but flagged + excluded from settlement).
  • At cycle close: only verified-paid orders settle (belt and braces).

This is build item 7B and lands before the first real cycle closes.
