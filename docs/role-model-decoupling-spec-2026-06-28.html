Role-Model Decoupling — Design Spec

PAS · drafted 2026-06-28 · status: FOR REVIEW (no code yet, beyond the shipped deletion guard)

One sentence: PAS already treats a person as one identity (one phone → one login → one id shared by both apps), but User.role is a single mutually-exclusive label that the merchant paths overwrite to MERCHANT and never restore — so the system forgets a store-owner is also a customer. This spec replaces the either/or label with "everyone is a customer; owning a store is an additive capability."

1. The problem

From the 2026-06-28 codebase scan (evidence cited inline):

  • One identity by id. Both apps call the same phone-keyed /auth/verify-otp; User.id == Merchant.id == Store.id == Branch.id (the owner's auth UUID). index.ts:7321, 7998, 8173, 8217; schema.prisma:211
  • But User.role is a single scalar — enum Role { SUPER_ADMIN, OPERATIONS, FINANCE, SUPPORT, MERCHANT, CONSUMER }, @default(CONSUMER). No "both". schema.prisma:131,1456
  • The merchant paths overwrite it in place and never restore it:
    • Every OTP login auto-flips role→MERCHANT if the phone matches any branch. index.ts:7463-7484
    • Merchant signup upserts role→MERCHANT. index.ts:7983 · 8592 ("user might exist as consumer")
    • The sync_merchant_data_robust() trigger re-stamps MERCHANT on every merchant write. fix_signup_sync_trigger_storename.sql:51,60
  • Precedent for the fix already exists: "is this person an admin?" was split off into a separate boolean User.isAdmin precisely so it survives role overwrites. schema.prisma:134; index.ts:7507 — merchant/consumer never got that treatment.
  • The one deliberate dual-role accommodation is notification recipient_role (separates a person's merchant vs customer inbox). schema.prisma:176; useNotifications.ts:54

3. Target model & principle

Principle: Identity is one row keyed by the auth id (already true). Everyone is a consumer by default and can always order. "Merchant" is a capability derived from owning a merchants/Store record — never a label that erases the consumer side. Mirror how isAdmin already works.

Decision point: (a) derive merchant-ness on the fly from "does a merchants row exist for this id" (no schema change, just stop overwriting role and stop gating on it), or (b) add an explicit User.isMerchant boolean (cheap to query, survives, symmetrical with isAdmin). Recommendation: (b) — explicit, fast, and self-documenting; role then stays at its base value (CONSUMER for almost everyone).

4. Concrete changes by surface

A · Stop overwriting role

  • JIT provisioning index.ts:7463-7484: drop the role:'MERCHANT' upsert; instead set isMerchant=true (or simply do nothing — the merchants row is the source of truth). Keep linking branch managers via store_staff.
  • Merchant signup upserts index.ts:7983, 8592 and the trigger fix_signup_sync_trigger_storename.sql:51,60: set isMerchant=true, do not change role (leave it CONSUMER).

B · Replace role='CONSUMER' gates with capability logic

  • Admin "new consumers" metric index.ts:9820 and customer DB UI CustomerDatabase.tsx:460: define "customer" as has placed an order (or NOT isMerchant-only), not role='CONSUMER'.
  • Audit every role === 'CONSUMER' / role !== 'CONSUMER' read repo-wide and re-express in capability terms.

C · Apps

  • Consumer app already ignores User.role (reads profiles only) — no change needed; a merchant ordering from another store "just works" once the role-gated server queries (B) are fixed.
  • Merchant app derives owner/manager from merchants/store_staff at login login.tsx:90-162 — already capability-based; just widen UserContext typing so a non-merchant isn't defaulted to MERCHANT. UserContext.tsx:82

D · Self-order policy

Decision: when consumer_user_id == branch.merchantId (ordering from your own store): block it ("you can't order from your own store") or allow it but exclude from "new customer"/settlement attribution. Recommendation: block in POST /order-requests — simplest and avoids self-settlement/coupon-on-self math.

E · Email seam

  • Prefer id-keyed upserts over email-keyed ones in merchant signup index.ts:8588; keep the auth/synthetic email stable and store the merchant's real/contact email in a separate field (e.g. Merchant.contactEmail) so it never collides with User.email @unique.

F · Notifications

  • Keep recipient_role (it's the right pattern) but add a server-side assertion/lint so no notification insert can omit it.

5. Phased rollout

  1. Phase 0 — stop the bleeding. DONE 2026-06-28 Deletion guard: merchants blocked from the consumer self-delete; consumer delete aborts (no auth-delete) if the User row can't be removed. index.ts /auth/delete-account
  2. Phase 1 — add the capability flag. Migration: User.isMerchant boolean default false. Backfill true for every id that owns a merchants row. Stop the role overwrites (A) and set isMerchant instead. No behaviour change yet for readers.
  3. Phase 2 — flip the readers. Re-express the role='CONSUMER' gates (B) in capability terms. Now dual-role people appear correctly in customer lists/counts.
  4. Phase 3 — restore the base role. One-time: set role='CONSUMER' for users currently MERCHANT who should be base-consumer (keep isMerchant=true). Verify no reader still depends on role='MERCHANT' for capability.
  5. Phase 4 — self-order + email seam (C/D/E/F) as independent hardening.
  6. Phase 5 — data cleanup. Repair the already-corrupted accounts (orphaned merchant Users from the old broken deletion; duplicate rows from the email seam) — separate one-off, gated.

6. Risks

  • Hidden readers of role. Some gate may rely on role='MERCHANT' for a real capability (e.g. an admin or merchant-only endpoint). Phase 1 keeps role intact while we add the flag, so we can migrate readers safely before Phase 3 restores the base value.
  • The merchants trigger is applied via Supabase SQL editor (not Prisma migrate) — must be re-edited in lock-step and re-applied to prod. fix_signup_sync_trigger_storename.sql
  • RBAC roles (SUPER_ADMIN/OPERATIONS/FINANCE/SUPPORT) also live on role — decoupling must not disturb staff/admin role handling (already partly shielded by isAdmin).
