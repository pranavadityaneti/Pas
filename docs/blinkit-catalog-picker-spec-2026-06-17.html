Catalog Picker — Design Spec

Phase 4, sub-project 2 · Branch feat/consumer-global-config-wiring · 2026-06-17 · Status: design locked, awaiting user review before plan

In plain English

Today you have 140,000 Blinkit products sitting invisibly in the database. No merchant can sell them, no customer can see them. This sub-project builds the screen that turns those products into things merchants can actually list in their shops:

  1. A merchant taps "Add products" in the merchant app.
  2. They browse a fast, searchable, filterable list of the 140k products (image, name, brand, pack size, MRP).
  3. They tap to select the products they want to sell ("12 selected").
  4. They tap "Configure" → a single screen lets them set their selling price and stock for each.
  5. They save. Those products now appear in their store; customers can browse and buy them.

Three safety nets are wired in at every layer (app, server, database) so misuse is impossible: (a) a merchant cannot charge more than the printed MRP; (b) a merchant cannot list food products without a valid FSSAI licence on file; (c) product images get silently copied off the competitor's CDN (cdn.grofers.com) onto your own Supabase bucket the first time a merchant lists a product.

1. Scope

In

  • Paginated, searchable, filterable catalog picker that scales to 140k Products.
  • Multi-select → bulk-configure UX (set selling price + stock per Product → save in one API call).
  • Three listing guards baked in at all three layers:
    • MRP ceiling (StoreProduct.price ≤ Product.mrp): app validation + API 400 + DB trigger.
    • FSSAI gate for food verticals (Vertical.requiresFssai = true): UI banner + Save disabled + API 403.
    • Lazy image re-host grofers.com → Supabase products bucket on first listing.
  • Rewrite of catalog-picker.tsx (screen body — the @lock'd FilterModal invocation and DEFAULT_FILTERS stay untouched).
  • Rebuild of ConfigureProductsModal (drop fabricated variants, drop direct Supabase reads).
  • Extend POST /merchant/store-products/configure with the guards.
  • New service apps/api/src/services/imageRehost.ts.
  • One DB trigger + one trigram index.

Out (deferred to later sub-projects)

  • Consumer veg/non-veg filter + dots (= sub-project 3).
  • Backfill veg flag for the 294 legacy products (= sub-project 3).
  • Sibling / variant grouping (UI shows one Product per row; StoreProduct.variant stays 'Standard').
  • Background job queue / async re-host worker (synchronous-but-failure-tolerant re-host on the save path is sufficient).

2. Decisions locked (this session)

#DecisionChoice
D1MRP-ceiling DB layerBEFORE INSERT/UPDATE trigger on StoreProduct that joins Product and raises on violation. Mirrors the spirit of the ₹0 CHECK (defense-in-depth), even though a same-row CHECK can't reference Product.mrp.
D2FSSAI strictnessSoft warning + hard server block. The picker shows a banner; the bulk-configure modal disables Save with an "Add FSSAI" CTA; the API returns 403 with {error:'FSSAI_REQUIRED'}.
D3Re-host failure modeKeep grofers URL, log to Sentry, save the listing anyway. The next merchant listing the same product retries. SafeImage (Phase 3) covers any genuine 404.
D4Listing UX flowMulti-select picker → bulk-configure modal. One API transaction per save. Drops the fabricated-variants modal entirely.
D5Variant storyOne Product = one StoreProduct row. variant='Standard'. Siblings/pack-size grouping is a future sub-project.
D6Pagination strategyKeyset on (createdAt DESC, id DESC). Stable under ingest, scales to 140k.
D7SearchTrigram GIN on Product.name + ILIKE in the API. Debounced 300ms on the client.
D8Re-host trigger locationServer-side, on the save endpoint, in-process (no queue). Tolerant of failure per D3.

4. New API endpoint — GET /merchant/catalog

Request

GET /merchant/catalog
Authorization: Bearer <merchant JWT>
?cursor=<opaque cursor base64({createdAt, id})>
?q=<debounced search 2+ chars>
?verticalId=<uuid>            ← user-facing "Category"
?categoryId=<uuid>            ← user-facing "Subcategory" (Tier2Category)
?brand=<string>
?isVeg=true|false|all
?limit=30                       (server clamps 1..50, default 30)

Response

{
  data: [
    {
      id, name, brand, mrp,
      image,                      ← may be grofers URL (will be re-hosted on first list)
      uom,                        ← pack size, e.g. "500 g", "1 pack (15 pcs)"
      isVeg,                      ← true | false | null (non-food = null = no dot)
      vertical: { id, name, requiresFssai },  ← drives the FSSAI banner
      category: { id, name }      ← subcategory
    },
    …
  ],
  nextCursor: <opaque base64> | null,   ← null = end of results
  hasMore: boolean
}

Logic

  • Auth: requireUser. Resolve the merchant's branch (the active branch for this user).
  • Where clause: 
    where: {
  source: { in: ['blinkit','live_sync','purchased_catalog'] }, // exclude null + 'zepto' (legacy)
  mrp: { gt: 0 },                       // no-MRP products can't be listed (can't ceiling-check)
  NOT: { storeProducts: { some: {       // exclude products this merchant's branch already lists
    branch_id: <merchant's branch>,
    is_deleted: false
  }}},
  ...(q ? { name: { contains: q, mode: 'insensitive' } } : {}),  // trigram-indexed
  ...(verticalId ? { vertical_id: verticalId } : {}),
  ...(categoryId ? { category_id: categoryId } : {}),
  ...(brand ? { brand: { equals: brand, mode: 'insensitive' } } : {}),
  ...(isVeg === 'true' ? { isVeg: true } : isVeg === 'false' ? { isVeg: false } : {})
}
  • Order: orderBy: [{ createdAt: 'desc' }, { id: 'desc' }] (matches the cursor).
  • Pagination: Decode cursor → add cursor: { createdAt_id } + skip: 1. Take limit + 1, then trim last and set nextCursor.
  • Include: Vertical: { select: { id, name, requiresFssai } }, Tier2Category: { select: { id, name } }.
  • Map to the response shape above.

Validation rules (client-side; mirrored server-side)

  • stock required, integer ≥ 0.

  • Save button disabled until every row is valid AND no FSSAI block.

7. Save endpoint — POST /merchant/store-products/configure

Extend the existing endpoint (apps/api/src/index.ts:10544). Body becomes:

{
  items: [
    { productId: uuid, price: number, stock: number }, …
  ]
}

Server logic (in a transaction)

  1. requireUser → resolve merchant + active branch. If no branch → 400. items must be a non-empty array (≤ 200) → else 400.
  2. Fetch all Products in items with their Vertical.requiresFssai + mrp. Any missing id → 400 with the offenders.
  3. MRP-ceiling check: any price > product.mrp → 400 with {error:'MRP_CEILING_VIOLATED', offenders:[{productId, price, mrp}]}.
  4. FSSAI check: if any selected product is food (requiresFssai) and merchant.fssaiNumber == null → 403 with {error:'FSSAI_REQUIRED'}.
  5. Write via createMany: {productId, branch_id, variant:'Standard', price, stock, active: price>0, is_deleted:false}. Conflict on the existing unique key (per branch + product + variant) → skip (idempotent re-listing).
  6. Trigger re-host with a bounded budget: Promise.race([imageRehost.rehostMany(productIds), sleep(5000)]). Whatever finishes within 5s is reported; outstanding ones are abandoned (the listing succeeds either way). Failures are logged to Sentry, do not roll back the listing.
  7. Audit-log the listing event (existing pattern). Respond {ok:true, listed:N, rehosted:M, rehostFailed:K}.

8. The lazy image re-host service

New file apps/api/src/services/imageRehost.ts.

export async function rehostOne(productId: string): Promise<'ok' | 'skipped' | 'failed'> {
  const p = await prisma.product.findUnique({ where: { id: productId }, select: { id:true, image:true }});
  if (!p?.image) return 'skipped';
  if (!p.image.includes('cdn.grofers.com')) return 'skipped';

  try {
    const res = await axios.get(p.image, { responseType: 'arraybuffer', timeout: 10_000, maxContentLength: 5 * 1024 * 1024 });
    const ext = inferExt(res.headers['content-type'] || '', p.image); // jpg|png|webp
    const path = `catalog/${productId}.${ext}`;

    const up = await supabase.storage.from('products').upload(path, res.data, {
      contentType: res.headers['content-type'] || 'image/jpeg',
      upsert: true,
    });
    if (up.error) throw up.error;

    const { data: pub } = supabase.storage.from('products').getPublicUrl(path);
    await prisma.product.update({ where: { id: productId }, data: { image: pub.publicUrl }});
    return 'ok';
  } catch (e) {
    Sentry.captureException(e, { tags: { area: 'imageRehost' }, extra: { productId, src: p.image }});
    return 'failed';
  }
}

export async function rehostMany(productIds: string[]): Promise<{ok:number; skipped:number; failed:number}> {
  // Limit concurrency to 4 to avoid overwhelming Supabase / our server
  const results = await pLimit(4, productIds.map(id => () => rehostOne(id)));
  return tally(results);
}

No new dependencies needed: axios is already a transitive dep; p-limit is small or we use a tiny inline limiter. The Supabase storage client is already imported (per existing upload code at index.ts:1257).

9. DB changes (trigger + index)

Trigger — trg_storeproduct_mrp_ceiling

CREATE OR REPLACE FUNCTION public.enforce_mrp_ceiling()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
  product_mrp double precision;
BEGIN
  SELECT mrp INTO product_mrp FROM public."Product" WHERE id = NEW."productId";
  IF product_mrp IS NULL THEN
    RAISE EXCEPTION 'Product % not found for StoreProduct write', NEW."productId";
  END IF;
  IF NEW.price > product_mrp THEN
    RAISE EXCEPTION 'MRP_CEILING_VIOLATED: price % exceeds Product.mrp %', NEW.price, product_mrp;
  END IF;
  RETURN NEW;
END $$;

DROP TRIGGER IF EXISTS trg_storeproduct_mrp_ceiling ON public."StoreProduct";
CREATE TRIGGER trg_storeproduct_mrp_ceiling
  BEFORE INSERT OR UPDATE OF price, "productId" ON public."StoreProduct"
  FOR EACH ROW EXECUTE FUNCTION public.enforce_mrp_ceiling();

Index — trigram GIN on Product.name

CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX IF NOT EXISTS idx_product_name_trgm
  ON public."Product" USING gin (name gin_trgm_ops);

Both applied via a tsx script using $executeRawUnsafe (matches the existing migration pattern). Verified with rolled-back transactions: positive (price ≤ mrp → row inserts) + negative (price > mrp → exception raised) on the trigger; EXPLAIN ANALYZE for an ILIKE before/after the index.

Live-data pre-flight: already verified — 0 existing StoreProduct rows violate the new trigger (SELECT count(*) FROM "StoreProduct" sp JOIN "Product" pr ON pr.id=sp."productId" WHERE sp.price > pr.mrp → 0). The trigger can be added without breaking any existing row.

11. Testing strategy

Unit (node:test via npx tsx --test) — same pattern as the Blinkit loader

  • src/services/imageRehost.test.ts — mocked axios + Supabase: ok / skipped (non-grofers) / failed; never throws.
  • src/merchantCatalog/cursor.test.ts — encode/decode round-trip; invalid cursor → null.
  • src/merchantCatalog/validate.test.ts — price ≤ mrp; price > 0; stock ≥ 0; FSSAI required when any item is food.

Integration / adversarial audit (read-mostly script, like the loader's audit)

  • scripts/audit_catalog_picker.ts — hits prod API with a test merchant: happy path, MRP violation, FSSAI missing, already-listed exclusion, cursor pagination round-trip, filter combinations, large search.

DB trigger

  • Rolled-back tx: positive (insert with price=mrp → ok); negative (insert with price=mrp+1 → exception). Both before deploy.
