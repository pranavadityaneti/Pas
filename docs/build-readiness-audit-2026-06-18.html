Why a build is required (not OTA-able)

An over-the-air update can only ship JavaScript. It physically cannot add native modules or change native entitlements/permissions — those live in the compiled binary. Since the last submitted build, both apps gained native-level changes:

The decisive one — expo-updates itself was added. That's the OTA runtime. Any build that predates it cannot receive OTAs at all. So a fresh build carrying expo-updates is the prerequisite for every future OTA. Combined with runtimeVersion: "appVersion" (an OTA only reaches a build of the same version string), the 1.1.2 / 1.2.5 binaries must exist before any 1.1.2 / 1.2.5 OTA can land.

The OTA plan (after builds are live)

Key point: if you build at HEAD, there is nothing to OTA right away — the binary is current. The OTA queue is for changes made after the build is cut.
  1. Cut + submit both builds (Consumer 1.1.2, Merchant 1.2.5) at c27cf617.
  2. Once they're live (or in TestFlight/internal track), any further JS-only fix becomes an OTA candidate. We'll list them as they arise and push eas update --branch production one app at a time, verifying each.
  3. Pre-OTA .env protocol (CLAUDE.md): OTAs (eas update) read the local .env (unlike builds, which use the eas.json profile env). Before each OTA: confirm EXPO_PUBLIC_API_URL=https://api.pickatstore.io, push with --clear-cache, grep the bundle to confirm the prod URL baked in. (Both .env are currently on prod — verified — but follow the step regardless.)
  4. runtimeVersion gotcha: an OTA only reaches builds of the same app version. After shipping 1.1.2/1.2.5, OTAs target those versions; users on older versions must update via the store.
