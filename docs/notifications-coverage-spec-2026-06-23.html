Notification Coverage — Spec & Phased Plan

#14 · lifecycle-only scope (Pranav, 2026-06-23) · branch feat/consumer-global-config-wiring = main
In plain English. Your apps already send most order notifications — about 24 of them are live (order confirmed, ready, cancelled, returns, etc.). This isn't a rebuild. Three things are left: (1) a small plumbing change so each app shows the right person's notifications cleanly (the "recipient role" switch that was always planned for "after June 6" — that time is now); (2) fixing a low-stock alert that's wired the wrong way (it never actually pushes to the merchant's phone); and (3) adding the 4 customer notifications that are genuinely missing from the money journey — payment failed, no store accepted your order, refund started, refund credited. Infrastructure is solid: NotificationService (apps/api/src/services/notification.service.ts) writes an in-app row and sends Expo push to all active devices, for both roles, inside the order transaction, deactivating stale tokens. It already sets recipientRole: 'merchant' | 'consumer' on every insert. ~24 dispatch sites are live across index.ts + scheduled-jobs.ts.

Two anti-patterns (solid-fix territory, per the No-Patches rule)

AP-1 — Low-stock is inserted client-side. apps/merchant-app/src/hooks/useInventory.ts:165–185 does two direct supabase.from('notifications').insert(...) calls (type INVENTORY, "(Manual)") when a merchant edits stock to 0/low. This bypasses NotificationService → no push fires and recipient_role is never set. After the read-cutover (below) these rows would disappear from the merchant inbox (they're neither 'merchant' nor 'consumer'). Fix: move the alert server-side into the manual-stock-edit API path (PATCH /merchant/store-products/:id) via sendMerchantNotification; delete the client inserts.
AP-2 — Apps read by a type-allowlist, not recipient_role. The consumer hook (useNotifications.ts) filters .in('type', CONSUMER_NOTIFICATION_TYPES) — a hardcoded 14-type list with an explicit comment: "once the recipient_role migration is live AND the API writes it, replace this allowlist." Both conditions are now true. The merchant hook filters only by store_id (no role scoping). This is the recipient_role cutover (your task #2) and is the clean foundation for #14 — after it, every new customer notification appears by role automatically with no allowlist edits.

3) Design decisions

#Decision
D1Do the cutover first (fold task #2 into Phase 1). Every new lifecycle notification then shows by role with zero allowlist maintenance.
D2Phase 1 is coupled: the cutover (AP-2) + the low-stock server move (AP-1) + the NULL backfill ship together. Shipping the cutover without AP-1 would silently drop manual low-stock alerts; shipping without the backfill would drop the 12 legacy merchant rows.
D3Cutover read change: consumer → .eq('recipient_role','consumer') (delete the allowlist constant); merchant → add .eq('recipient_role','merchant') alongside its store_id filter.
D4Backfill: UPDATE notifications SET recipient_role='merchant' WHERE recipient_role IS NULL — all 12 NULLs are confirmed merchant types. Idempotent, reversible (snapshot the ids first).
D5REFUND_CREDITED sub-decision: (a) ship only REFUND_INITIATED now (easy, at refund creation) and defer REFUND_CREDITED until a refund.processed webhook is added; OR (b) add the refund webhook in Phase 2. Recommend (a) — initiated is the high-value "your money is on the way" message; credited can follow once the webhook is subscribed in the Razorpay dashboard. (Confirm at Phase 2 start.)
D6No quiet-hours / digest batching in this scope — every notification is transactional + user-triggered, so volume is naturally low. Defer batching to a later phase if push fatigue ever shows up in data.
D7Reuse existing patterns exactly: same NotificationService call shape, same link/metadata/referenceId contract, same routeForNotification deep-link helpers. No new infrastructure.

4) Phase 1 — Role-routing foundation (coupled)

DB + API + both apps Ships as: one DB backfill · one API add (server-side low-stock) · consumer OTA · merchant OTA.

  1. 1a — Backfill (DB). UPDATE notifications SET recipient_role='merchant' WHERE recipient_role IS NULL. tsx script with a rollback JSON of the 12 affected ids. Verify 0 NULLs after.
  2. 1b — Consumer read cutover (app → OTA). In apps/consumer-app/src/hooks/useNotifications.ts: replace .in('type', CONSUMER_NOTIFICATION_TYPES) with .eq('recipient_role', 'consumer'); delete the CONSUMER_NOTIFICATION_TYPES constant + its comment. Keep the realtime subscription + read/markRead untouched.
  3. 1c — Merchant read cutover (app → OTA). In apps/merchant-app/src/hooks/useNotifications.ts: add .eq('recipient_role', 'merchant') to the fetch (line ~55) and the realtime/markRead reads as needed.
  4. 1d — Low-stock anti-pattern (app removal + API add). Delete the two client-side inserts in useInventory.ts:165–185. Add a server-side low-stock dispatch in the manual-stock-edit API path (PATCH /merchant/store-products/:id): when an edit takes stock to 0 or ≤5, call sendMerchantNotification({ type:'LOW_STOCK', ... }) (sets recipient_role + push). Mirror the existing order-decrement LOW_STOCK at index.ts:3427.
  5. 1e — Audit (mandatory, rule #3). Confirm: 0 NULL rows; consumer inbox shows exactly its role rows (no merchant bleed for dual-role accounts); merchant inbox unaffected by consumer rows; the manual low-stock now pushes; the 14 previously-allowlisted types all still appear.

5) Phase 2 — Lifecycle gap scenarios

API only → EB deploy All four are server-side dispatches; they appear by role automatically (post-Phase-1). No app change, no OTA.

  1. 2a — PAYMENT_FAILED. In the payment.failed webhook branch (index.ts:781): resolve the consumer userId (from the Razorpay order notes / orphaned-payment record), then sendConsumerNotification. Edge: if no order/user resolves, log to Sentry and skip (don't throw).
  2. 2b — ORDER_REQUEST_EXPIRED. In expireStaleOrderRequests (scheduled-jobs.ts:571): for each request flipped to EXPIRED, sendConsumerNotification to consumer_user_id. Pre-payment → "you haven't been charged."
  3. 2c — REFUND_INITIATED. At each tryRazorpayRefund success (SLA auto-refund + issue-PATCH refund): sendConsumerNotification with the amount + "5–7 business days." Skip on simulated/stub refunds (or label them).
  4. 2d — REFUND_CREDITED (per D5). If approved: add a Razorpay refund.processed webhook branch → sendConsumerNotification. Otherwise deferred.
  5. 2e — Audit (mandatory). Each new dispatch: correct userId resolution, fail-soft on missing user, no transaction rollback risk (push is outside tx), no duplicate sends (idempotent per order/refund).

6) Phase 3 — Verify, deploy, OTA

  1. tsc --noEmit on api + both apps after each phase; per-phase line-by-line audit.
  2. Phase 1: run the backfill script (gated) → npm run build + eb deploy (server low-stock) → consumer OTA + merchant OTA (the read cutovers + client-insert removal).
  3. Phase 2: eb deploy only.
  4. End-to-end smoke per scenario (trigger each, confirm in-app row + push + correct inbox + deep-link route).
