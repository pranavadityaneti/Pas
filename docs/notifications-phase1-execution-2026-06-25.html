3) The audit — no misses, no leaks, not a patch

The one failure class this change could introduce: after the cutover, the apps read notifications by recipient_role, so any notification written without that tag becomes invisible. The audit traced every write and read across the whole repo to prove that can't happen.
CheckFinding
Every write is tagged✓ Across all of api/src there are exactly two notification-creating paths (sendMerchantNotification, sendConsumerNotification) and both unconditionally set recipientRole. No raw insert, no createMany/upsert, no bypass — in the API or the crons.
Why it's a fix, not a patch✓ The class is closed for all past rows (backfill) and all future rows (no code path can write an untagged notification). Root cause, every surface.
Every read is handled✓ admin-web touches the table nowhere. Both apps read it only via the two hooks (fetch + realtime + mark-all role-scoped; mark-as-read-by-id intentionally unscoped — it acts on a row already in the scoped list).
No collateral damage✓ Order-driven LOW_STOCK (untouched) already routes through the service → stays tagged merchant. Backfill touched only the 12 NULL rows; the 82 consumer + 295 existing merchant rows untouched.
Latent bug fixed in passing✓ Both apps' "mark all read" was unscoped — a dual-role account marking-all in one app marked the other role's rows read too. Now role-scoped.
Fail-closed✓ A NULL/missing recipient_role is treated as "not mine" in both realtime guards — a mistagged row can never leak into the wrong inbox.
Honest caveat — one thing not end-to-end fired. The server-side low-stock dispatch is verified by code + compile + deploy, but I did not trigger it against a real merchant's inventory (that's a live side-effect on a real store). Its first real proof is the next genuine manual stock-out. Likewise the realtime guard relies on Supabase delivering recipient_role in the INSERT payload — guaranteed by-design (INSERT events carry all columns), and the in-app list works regardless, but not live-fired.

4) Plain English — what changes for people

  • Nothing breaks, immediately. Users still on the old build see exactly what they saw before — the backfill only added a tag, it removed nothing. Users who get the OTA switch to the cleaner role-based behavior.
  • Merchants: the inbox is scoped to merchant notifications by identity (not a store-ID workaround), so a merchant who also shops as a customer never sees customer notifications bleed in. And a manual stock-out / low-stock now pushes to their phone and looks identical to the automatic order-driven alert — before, it was created silently with no push and would have vanished after this change.
  • Customers: the inbox is scoped to consumer notifications by identity. The real win is future-proofing — every new customer notification (Phase 2's payment-failed, refund-started, etc.) lands in the right inbox automatically, with no app update.

5) What's not done + safety nets

  • Phase 2 (next, API-only / EB): the 4 lifecycle gaps — PAYMENT_FAILED, ORDER_REQUEST_EXPIRED, REFUND_INITIATED, and REFUND_CREDITED (this last needs a Razorpay refund.processed webhook — decision D5). They'll show by role automatically now that the cutover is live.
  • Deploy-window note: a manual stock-out in the brief gap between the EB deploy and a device receiving the OTA could transiently double-alert or briefly not alert — harmless, self-healing.
  • Rollback if ever needed: restore the 12 ids in apps/api/scripts/_notif_recipient_role_backfill_rollback_2026-06-25.json to NULL, and re-publish the prior OTA bundles. The EB deploy reverts with one redeploy.
