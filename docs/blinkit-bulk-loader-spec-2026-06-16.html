1. Purpose & success criteria

Load the purchased Blinkit dataset (141,405 rows, 165 MB CSV) into the PAS master catalog — i.e. Product rows only. These are catalog entries, not inventory: they become pickable by merchants (sub-project 2) and surface to customers only when a merchant lists one (creates a StoreProduct with price/stock). This load exposes nothing to customers.

Success = a clean, correctly-categorized, veg-derived catalog where every loaded row has a real PAS vertical + category, a preserved sibling fingerprint, an MRP, and a suggested selling price — ready for the picker. Unmappable rows are skipped (not polluted in), and the run is safe, resumable, and fully reported.

3. Architecture & execution

A single TypeScript CLI: apps/api/scripts/load_blinkit_catalog.ts, run with npx tsx from apps/api using the production DATABASE_URL (the same way every migration this session ran). It streams the CSV with csv-parse in stream mode (constant memory — never the whole 165 MB in RAM), transforms each row, and upserts in batches via the Prisma client.

FLAGS
  --file <path>        path to Blinkit Data.csv (required)
  --dry-run            run the full pipeline, write NOTHING, print the report (incl. category hit-rate)
  --limit <N>          process only the first N data rows (for test loads)
  --batch-size <N>     rows per upsert batch (default 500)
  --skip-existing      preload existing source_product_ids into a Set and skip them (fast resume)

RUN ORDER (enforced by us, not the script)
  1. --dry-run on the full file        → review skip-rate + sample mapped rows
  2. --limit 100 real load             → spot-check 100 inserted rows
  3. full load                         → only on Pranav's explicit go (migration rule)

No HTTP endpoint, no queue, no worker. The existing in-memory POST /products/bulk-import-json is explicitly NOT used (it buffers the whole payload).

4. Per-row pipeline

for each CSV row:
  1. PARSE      14 columns + data_dump JSON
  2. CATEGORY   lookup (category, subcategory) in CategoryMapping
                  → vertical_id + category_id
                  unmapped / pending / NULL  →  SKIP (count: skipped_unmapped)
  3. VEG        is_food = vertical.requiresFssai
                  non-food            → isVeg = NULL  (unknown, no dot)
                  food + non-veg sig  → isVeg = false
                  food + no signal    → isVeg = true
  4. MAP        build the Product row (see §5), incl. extraData
  5. BUFFER     push to batch; flush at batch-size (upsert on source_product_id)

6. Veg-derivation rules

Blinkit has no veg flag, so the loader derives one. Tri-state: true / false / NULL(unknown).

OPEN DECISION — where to store it. Recommended: a new Product.isVeg Boolean? column (added by a small additive migration before the load). Reason: sub-project 3 builds a veg/non-veg filter over 141k rows — a real, indexable column is far better than querying a JSON field.
Trade-off: the consumer useProducts hook (@lock) currently reads extra_data?.isVeg. Switching it to read the column is part of sub-project 3 (the veg UI work) — fine, because loaded products aren't customer-visible until a merchant lists them (sub-project 2), so no veg dot renders from this data until then anyway. Alternative: write extraData.isVeg (no migration, matches today's read, but JSON-filtering later). Pranav to confirm.
is_food   := resolved vertical.requiresFssai == true
             (food verticals: Grocery & Kirana, Fresh Items,
              Bakeries & Desserts, Restaurants & Cafes)

NON-VEG if is_food AND (
     subcategory IN {"Chicken, Meat & Fish", "Frozen Non-Veg", ...}
  OR name/subcategory matches any of:
     chicken, mutton, fish, prawn, shrimp, seafood, crab, meat,
     egg, anda, pork, bacon, ham, sausage, salami, keema, fillet,
     lamb, tuna, squid, octopus
)
VEG     if is_food AND no non-veg signal      (default-veg, per decision)
UNKNOWN if NOT is_food                        (electronics, pet, fashion → no dot)
Tunable + reviewable. The keyword/subcategory list is data-driven and the --dry-run report prints the veg/non-veg/unknown distribution + a sample of each, so we tune the list before the real load. Egg is treated non-veg (Indian convention).

7. Idempotency, resume & batching

  • Upsert keyed on sourceProductId — a re-run inserts new rows and updates existing ones; it never duplicates.
  • Batches of ~500, per-batch commit. Never one giant transaction (the forlater's 60s-tx warning). A crash at row 90,000 loses nothing committed.
  • Resume is just "run it again." --skip-existing preloads existing ids to skip re-processing for speed.
  • In-file dedup: Blinkit product_id has 0 dups (verified), but the loader still guards against a duplicate id within the file (counts skipped_dup).

8. Reporting & observability

Live progress every N rows, and a final summary:

BLINKIT LOAD — summary
  total rows read .......... 141,405
  loaded (insert+update) ... <n>
  skipped — unmapped cat ... <n>   (category hit-rate = loaded / mappable)
  skipped — dup in file .... <n>
  errors ................... <n>   (+ first 20 samples)
  veg / non-veg / unknown .. <a> / <b> / <c>
  per-vertical counts ...... { ... }

The --dry-run produces this exact report writing nothing — so the category hit-rate and veg distribution are reviewed before any prod write.

9. Error handling

Per-row errors are caught, logged, and counted — a malformed row never aborts the run. A failed batch is retried row-by-row to isolate the offender, which is logged (with its product_id) and skipped; the run continues. The summary surfaces the total error count + samples so nothing fails silently.

10. Reversibility

Every row is tagged source='blinkit'. Before any merchant lists a Blinkit product, a full rollback is a single statement:

DELETE FROM "Product" WHERE source = 'blinkit';   -- safe pre-listing

(Cascades to ProductImage; no StoreProduct exists yet for these.) This dovetails with the deferred "delete old source='zepto' data" cleanup, which happens after this load is verified.

11. Testing plan

  1. Dry-run, full file → review category hit-rate, veg distribution, and a sample of mapped rows. Tune the veg keyword list / flag any category-mapping gaps.
  2. --limit 100 real load → spot-check 100 inserted rows in the DB: category/vertical correct, veg sensible, mrp + suggestedPrice + uom + image URL + sibling fields all present and right.
  3. Idempotency check → re-run the same 100; assert 0 duplicates, counts stable.
  4. Full load → only on explicit go; verify final counts + per-vertical + veg distribution against the dry-run prediction.

13. Open risks & notes

  • Category hit-rate is unknown until the dry-run. If a large fraction skips (mapping gaps beyond the known ~325 NULL + 13 pending), we extend CategoryMapping before the real load rather than loading a partial catalog.
  • Veg false-positives. Default-veg means an unflagged non-veg item (e.g. a sauce with fish extract) could show a veg dot. The dry-run distribution + keyword tuning is the mitigation; absolute correctness needs human spot-review of the non-veg-adjacent subcategories.
  • quantity parsing is best-effort. Free-text units ("1 kg", "6 x 100 g", "per piece") won't all parse cleanly; raw value is always kept in uom so nothing is lost.
  • Grofers image URLs may already be expiring. Acceptable: customers never see them (lazy re-host on listing), and SafeImage covers the admin picker.
  • This is a prod write. Dry-run → limited load → explicit go → full load. Rollback is one DELETE by source.
